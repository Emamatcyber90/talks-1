Website authentication methods and UX

Part 1
Login:

Login

  • Use email addresses instead of usernames for login.
    Most users only have one.
  • If your site offers user profiles, let them choose
    (and change) their visible name freely.
  • Allow both for login, reduces login errors
  • Also allow the user to change their email address
    (no one is on aol.com any more)

Part 2
Enter your email address:

This should be easy, right?

Confirm your email address:

  • Why? The user did just enter it the second before.
  • What do many people do? Copy & paste.

This doesn’t help at all to check the validity of the entered address.

Bonus part

Usability special-specialist at work

Right-click/copy & paste blocker in JavaScript.

Never, not under any circumstances, disable or overwrite functions of the user’s computer or make them behave any differently.

Part 3
Email address validation

Email address validation

Email address validation is only of limited use.

foo@bar.baz is still considered valid in most cases.

It only gives feedback about the general structure of the input (e.g. contains “@”), but doesn’t tell anything about the existence of a mail address.

Checking the DNS record for a domain

  • Only tells you if mail server exists for this domain.
  • It's a bit more "expensive" (requires external request)

google.com has an MX record, yet most likely you rather mean gmail.com

All of this is too much work, and we’re still not sure, if the email address exist.

Assisting the user

Tools like mailcheck make suggestions for possible spelling mistakes:

user@hotnail.con  =>  user@hotmail.com
user@tonline.de   =>  user@t-online.de
user@gxm.de       =>  user@gmx.de

Important: Always make suggestions maybe the address really exists and is no spelling mistake

The only reliable way to check an
email address is sending a mail.

Part 4
Enter your password:

Passwords suck.

Passwords suck.

We’ve now been preacing for decades to use safe passwords, still the most common passwords are:

123456          password          qwerty
abc123          111111            iloveyou
123123          admin             letmein
monkey          password1

So we spoon-feed our users to use safer passwords:

Your password must be between 8 and 14 characters long and contain at least one uppercase and lowercase character, one number, one special character (§$%&/_-) but not (@# or +), an emoji, your zodiac sign and blood type …

This is incredibly frustrating for users,
as they “just” want to use your service.

Confirm your password:

  • The password entered before is not shown
    (for good reasons)
  • High potential for typos/entering the password differently

= Frustration ⨉ 2

Some people will decide to leave your site at this point.

Improving the password situation

  • Offer a password strength indicator
  • Offer guidance to create strong memorable passwords (e.g. FAQ)

  • Secure passwords don't have to be cryptic and hard to remember (XKCD #936):

    "folly wood bruise cabin mad magpie" is more secure than "8%9B>a3yeHj#;1q" (77bit vs. 75bit entropy)
Sources: Diceware Passphrase Generator • Strength Test • The Password Meter

Part 5
"Securitry" questions

"Securitry" questions

Some questions are not unambiguous and can unintentionally lead to wrong answers:

Q: “What was your first car?”
A: Volkswagen? VW? Golf?

"Securitry" questions

With the ubiquity of social media profiles and Google, most of them should be easy to find out:

  • “Where did you go to school?”
  • “What’s your pet’s name?”
  • “What’s your mother’s maiden name?”

Those questions don’t increase security at all, their use is highly questionable.

Part 6
Ideas for authentication

One Time Passwords (OTP)

  • Generate a secure authentication token and mail it to the user.
  • User clicks link in email, token keeps him signed in with the website.
  • If you allow the user to sign in with multiple devices, show a list of registered devices and allow deactivation (see Dropbox for example).

One Time Passwords (OTP)

Pro:

  • Verified email address for a user
  • No password to remember
  • Works great for mobile devices

Contra:

  • Leads the user away from the website at first (may impact conversion rate)
  • Security issue on shared devices or public spaces (Internet café)
Allow the user a wide range of freedom on your site using anonymous sessions (e.g. to put items in cart), creates incentive to log in later

OAuth aka. "social login"

Pro:

  • Delegation of login and authentication mechanisms
  • Wide range of services exist (Twitter, Facebook, GitHub, Google, Yahoo!, etc.)

Contra:

  • Delegate login and authentication mechanisms (dependency on third party service, may collect data about your users)
  • Multiple services may confuse the user:
    »Which service did I use to sign in?«
  • What happens if a user switches the OAuth provider?

Mozilla Persona

»Mozilla Persona lets you sign in to websites with any email address in as few as two clicks, without the need to create a new password.«

Mozilla Persona

Pro:

  • Better privacy: Only the verified email address and a token is sent to the website
  • User only needs the password for their email account
  • Any service can become an identity provider (main providers: Mozilla, Yahoo!, Google)
  • Based on open web technologies

Contra:

  • No widespread use yet

Two factor authentication

Using SMS or key generator app

Two factor authentication

Pro:

  • Highly increased security
  • Great for services requiring sensitive user information (bank account, credit cards)

Contra:

  • Additional step, sign-in takes longer
  • Requires a second device (e.g. mobile phone, tablet)
  • Requires mobile/Internet connection
  • Very unhandy using the same mobile device to log in
    (Copy/pasting security code from SMS app, etc.)

Gather metrics

Gather metrics

Analyze your registration and login process:

  • How many potential users do you lose on the way during sign-up? (Especially interesting for multi-step-processes)
  • How many times is the registration page reloaded per user on sign-up?
  • How many times is the login page reloaded per user?
  • How many successful/failed logins occur?
    (Also helpful regarding website security)

Conclusions

Conclusions

  • Break with bad UX habits, because “everybody does it that way”
  • Make it as easy and comfortable for a user to sign-up to your website
  • Make it as secure as possible for a user to use your website
  • Think about the best authentication methods matching your business
